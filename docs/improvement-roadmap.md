# 补充建议审阅与路线图

> 审阅日期：2026-06-14  
> 来源报告：`E:\14.小技巧\14.购车APP\car-buying-calculator_补充报告.md`

## 总体判断

补充报告的方向整体合理。它围绕三个核心目标展开：扩大可覆盖的购车场景、降低工具使用门槛、提升后续维护质量。这与项目当前定位一致：把销售报价里混在一起的费用拆开，让用户能核算、能比较、能复盘。

需要注意的是，报告中涉及购置税、以旧换新、地方补贴、保险价格等政策或市场价格的建议，不能直接作为固定规则写死。此类内容必须保留官方公告、当地细则或保险公司报价的核验入口。

## 采纳建议

### P0：发布和文档

- 启用 GitHub Pages，让用户无需下载即可使用。
- README 保留截图、徽章、使用方式和在线体验入口。
- README 持续维护更新记录，记录每一次功能、口径、文档和安全改动。

### P1：低成本高收益功能

- 数据导入/导出：解决 localStorage 无法跨设备迁移、清缓存丢失记录的问题。
- 结果分享：优先做文本摘要复制，再考虑移动端 Web Share API。
- 政策参数外部化：先抽成集中 `POLICY` 对象，后续再拆到 `policy.json`。
- 输入校验增强：对指导价、税率、异常费用、补贴金额做即时提示。

### P2：核心场景扩展

- 贷款/分期计算：这是最大功能缺口，应支持首付、贷款本金、期数、年化利率、月供、总利息、金融服务费和全款/贷款差额。
- 保险费用估算：可做交强险、车损险、三者险、座位险拆分，但必须标注“仅供估算，实际以保险公司报价为准”。
- 浅色/深色主题：当前 CSS 变量基础较好，适合在功能稳定后补充。

### P3：体验增强

- 对比图表：在记录较多时提升扫描效率，优先使用零依赖 CSS 图表。
- 购车清单：把计算器扩展为购车流程助手，但优先级低于费用核算本身。
- 无障碍改进：补齐 label、aria-live、键盘操作和对比度检查。

### P4：工程拆分

当前单文件结构仍可接受。等导入导出、贷款、保险等功能增加后，再拆分为 `index.html`、`style.css`、`app.js`、`policy.js` 更合适。拆分时需要同步更新 `sw.js` 缓存清单。

## 调整意见

- 报告中“国内超过 60% 用户选择分期付款”的表述需要来源支撑；README 中不应直接引用未经核验的比例。
- GitHub Pages 链接只有在 Pages 实际启用后，才能在 README 中标注为“在线体验”；启用前应写成预计地址或部署说明。
- 以旧换新补贴金额、报废/置换条件、地方补贴规则应避免写死到 README，建议在代码中抽象为可更新配置，并在页面提示用户以官方公告为准。
- 保险估算不能承诺准确报价，只能作为谈价和识别异常费用的参考。
- 代码结构拆分不宜早于功能稳定，避免为拆分而拆分。

## 建议推进顺序

1. GitHub Pages 和 README 完善。
2. 报价记录导入/导出。
3. 结果文本分享。
4. 政策参数集中配置。
5. 贷款/分期购车计算。
6. 保险费用估算。
7. 输入校验、主题切换、图表、购车清单、无障碍。
8. 单文件拆分和测试补齐。
